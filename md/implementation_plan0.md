# LepoProxy 代理软件需求分析与技术建议书

您提供的《LepoProxy 代理软件设计与需求规格说明书》非常详尽且专业，整体逻辑清晰，功能划分也很合理。基于 Tauri + Rust + sing-box 的选型非常适合开发高性能、低内存占用的现代桌面代理客户端。

在正式编写代码和修改文件之前，我仔细阅读了您的需求，并从架构设计、技术实现难点以及后续的开发步骤上，为您提供以下**技术评估与建议**，供您审阅。

## 💡 架构设计与技术建议

### 1. Tauri (Rust + JS) 职责划分
为了保证软件的性能和安全性，建议明确前端（JS/HTML/CSS）与后端（Rust）的职责：
*   **前端 (HTML/CSS/JS)**：专注于 UI 渲染、动画交互、状态展示（连接状态、节点延迟等）、以及向后端发送命令。
*   **后端 (Rust)**：
    *   **核心进程管理**：负责启动、停止、监控 `sing-box.exe` 进程。
    *   **系统级操作**：修改 Windows 系统代理（通过修改注册表）、管理开机自启（写入启动项或注册服务）、解除 UWP 应用网络隔离。
    *   **文件与网络操作**：订阅的拉取 (`SubGet`)、解析与 JSON 转换 (`SubTrans`)，以及本地配置文件的读写。

### 2. 关键难点与解决方案

#### A. Tun 模式与权限问题 (UAC)
*   **难点**：开启 Tun 模式（虚拟网卡）以及修改某些系统网络设置**必须需要管理员权限**。
*   **建议**：
    *   **方案一**：整个 Tauri 应用以管理员权限运行（会产生 UAC 弹窗）。这是最简单的实现方式，但每次启动都需要用户授权。
    *   **方案二（推荐）**：主程序以普通用户权限运行。当用户首次开启 Tun 或设置相关功能时，通过 Rust 提权调用独立的脚本或服务去安装驱动/启动核心，或者将核心网络服务注册为 Windows 后台服务，前端通过 IPC 或本地 Socket 与其通信。考虑到开发成本，初期可以先采用**方案一**，后期再优化为服务分离。

#### B. UWP 应用网络隔离 (Loopback)
*   **难点**：Windows 的 UWP 应用（如微软商店、邮件等）运行在沙盒中，默认禁止将流量发送到本地环回地址（127.0.0.1），这会导致开启系统代理时这些应用无法联网。
*   **建议**：在 Rust 后端集成解除 UWP Loopback 限制的逻辑。可以调用 Windows API (`NetworkIsolationEnumAppContainers` 和 `NetworkIsolationSetAppContainerConfig`)，在开启代理时自动为所有 UWP 应用解除隔离。

#### C. 订阅解析与转换 (SubTrans)
*   **难点**：常见的订阅格式（Base64, V2rayN 格式, Clash YAML 等）繁杂，需要将其统一转换为 sing-box 复杂的 JSON 路由和出站格式。
*   **建议**：尽量在 Rust 端实现订阅解析，利用强大的正则和序列化库（如 `serde_json`）。对于各种协议（Vmess, Vless, Trojan, SS），我们需要构建对应的 sing-box outbound 模板。

#### D. 内置节点限时机制 (30分钟限制)
*   **难点**：需要记录用户使用内置节点的时长，并在超时后实现**无缝切换**。
*   **建议**：
    *   使用前端或 Rust 后端的定时器记录连接时长，并将累积时间持久化保存（如存入本地 JSON 配置文件）。
    *   “无缝切换”在底层意味着需要重新生成 sing-box 的配置文件并热重载或重启进程，可能会有瞬间的连接中断。我们可以通过 sing-box 内置的 `urltest`（延迟测试自动切换）或 API 动态更新出站节点来尽量减少断流感。

#### E. 节点延迟测试 (Ping)
*   **要求**：节点列表需要展示延迟并以红黄蓝绿区分。
*   **建议**：不要使用传统的 ICMP Ping（因为很多代理服务器禁 Ping），应该使用**基于代理连接的真连接测试**。可以通过 Rust 发起经过对应节点的 HTTP 请求（如访问 `http://cp.cloudflare.com/generate_204`），计算响应时间。

## 🗺️ 建议的分阶段实施计划

为了保证项目稳步推进，我建议将开发分为以下几个阶段：

### 阶段一：静态界面完善与基础框架搭建
1.  完善前端 `index.html`，补充完整的 UI 布局（主界面、节点选择遮罩、设置面板）。
2.  实现前端的 CSS 样式和交互动效（连接动画、面板切换、深浅色模式切换）。
3.  初始化 Tauri Rust 后端，打通 JS 与 Rust 的 IPC (进程间通信) 桥梁。

### 阶段二：核心业务逻辑 (Rust) 与 sing-box 对接
1.  在 Rust 端实现启动/关闭 `sing-box.exe` 的功能。
2.  实现基本的系统代理（System Proxy）开关逻辑。
3.  实现一套基础的 sing-box JSON 模板，允许前端传入服务器参数生成配置文件并运行。

### 阶段三：订阅、分流与高级功能
1.  实现 `SubGet`（拉取订阅）和 `SubTrans`（订阅解析转 sing-box JSON）。
2.  实现 `NodeSplitting`（直连、全局、规则分流）的配置动态生成。
3.  实现节点延迟测速功能，并同步渲染到前端。
4.  实现 Tun 模式和 UWP 解除隔离功能。

### 阶段四：完善特性与细节打磨
1.  实现内置节点的 30 分钟配额与自动切换逻辑。
2.  实现开机自启、托盘菜单 (System Tray)、深浅色主题跟随系统等。
3.  全局状态保存与读取（用户偏好设置持久化）。
